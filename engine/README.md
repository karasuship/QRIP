# engine — 検証エンジン（計算・可変の側）

思想（`CONSTITUTION.md` / `docs/philosophy/`）から物理的に分離した「計算」の側。
方針は `decisions/0006`。「思想は固定・計算は可変」をフォルダで守る。

## 動かし方（重要）

- **`python` は使わない。** このPCの `python` はMicrosoft Storeのスタブ（実体なし）。
  **`py`**（Python Launcher, 3.9）を使う。例：`py engine/heart.py`
- 標準ライブラリのみ（pandas等の依存なし）。最初から重い仕組みを入れない。

## 構成

- `fetch_data.py` … S&P500 日次を取得し `data/sp500_daily.csv` に保存（Yahoo `^GSPC` 調整後終値＝配当込みに近い・約30年）。取得後に素性を目視表示（地雷⑦）。
  - **データは git 管理しない**（`data/` は .gitignore）。各自 `py engine/fetch_data.py` で取得。再配布の権利問題も避ける。
- `heart.py` … 検証の心臓①。1仮説「ピークから-X%下落で買い、H日保有→将来リターン分布」を、**ベースライン（任意の日に買う）と比較**し、**OOS分割**する。条件を差し替えれば同じ形でどの仮説も検証できる。
- `archive/` … 検証結果の書庫。1発見1ファイル、素性ラベル付き（信頼設計＝decisions/0006）。成功も失敗も残す（地雷②）。

## ロードマップ（構築手順書 C→G、下から積む・地雷⑥）

1. ✅ 心臓①（1仮説＋ベースライン＋OOS）
2. 仮説リストの外出し（YAML/CSV、各仮説に因果の理由＝地雷①）
3. ループ（リストを上から自動検証）
4. 書庫＆重複検知（指紋で既出判定）
5. OOSの厳格化（行分割でなく地合い/イベント単位）＋多重検定（試行回数記録）
6. 多市場（TOPIX・各国・セクター）で独立イベントを増やす

## 既知の限界（最初の検証で露呈）

- 重複ウィンドウが見かけの精度を水増しする（延べ日数 ≠ 独立イベント数）。
- 単一市場・約30年では、深い下落の独立イベントが3〜4回しかない。
- 行ベースのOOS分割は地合いを跨ぐと無意味（深い下落は前半に偏る）。
- → だから「多市場・イベント単位・OOS厳格化」が次の必須課題。
