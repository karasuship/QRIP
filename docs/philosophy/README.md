# 記録システムの地図

QRIP の「思想・決定・知識」をどこに・どの濃さで保存し、いつ参照するか。**これが一貫した土台**。新しい情報が出たとき、ここを見れば「どこに入れるか」が決まる。迷いを無くしてぶれを防ぐためのファイル。

## 4つの階層

| 階層 | 場所 | 役割 | 記録の濃さ | 読まれ方 |
|---|---|---|---|---|
| **熱** | `CONSTITUTION.md` | 不変の芯。毎回守る判断基準 | 結論（最大圧縮） | 毎回自動で読む |
| **温** | `decisions/` | 「何を・なぜ・どう決めたか」の個別判断 | 思考圧縮 | 関連時に引く |
| **冷** | `docs/philosophy/` | 思想の出発点と、育っていく核 | 思考圧縮 | 立ち返るとき開く |
| **個** | Claude のメモリ（リポジトリ外） | Claude の運用規律 | 結論 | Claude が毎回読む（相方からは見えない） |

冷の層の内訳：
- `00-全文脈マスター.md` … **起源の記録（凍結）**。原則改変しない。最良の原典。
- `01-二次的な核.md` … **育つ思想（追記ログ）**。憲法に上げるほどではないが大事な核を、出るたびに積む。

## 分類ルール（新しい情報はどこに入れるか）

- 不変・毎回守る判断基準 → **憲法**（滅多に増やさない。増やすには改正手続き＝明示・理由・合意）
- 思想の核だが毎回は読まない・立ち返り用 → **冷アーカイブ `01`**
- 「何を・なぜ決めたか」の個別判断 → **decisions**（重い決定のみ。粒度は `decisions/README.md`）
- 計算・実証的な発見（法則・係数・検証結果） → **憲法に入れない。** 検証書庫へ（下記ルーティング表）。「思想は固定・計算は可変」より、計算を憲法化しない
- 可変の現状・段取り・技術選定の状態 → **`CLAUDE.md`**

## 記録の濃さ（3段階）

要約・思考圧縮・生ログは別物。階層ごとに使い分ける。

- **③要約（結論だけ）** … 憲法。芯だけでいい。「なぜ」は下の層が持つ
- **②思考圧縮（経路を残す）** … decisions・冷アーカイブの主役。「何を主張し→誰がどう反論し→どう修正し→なぜこう決めたか」を辿れる形。言い回し・重複・脱線は削るが、思考の筋は残す。**ただの要約にしない**
- **①生ログ（丸ごと）** … 控えの底。読まないが、原典・再抽出材料・証拠として保持

**生ログの扱い**：Claude Code とのセッション記録はこのPC内に自動保存されている（QRIP の開発会話の生ログはそこに残る）。節目の重要な議論で「丸ごと残したい」ものが出たら、`docs/philosophy/transcripts/` に能動的に退避する（必要が出たときに作る。今は予約のみ）。

## ガバナンス（誰が記録を決めるか）

- **Claude は、核となる思想がアーカイブ・憲法・decisions に加わるべきだと気づいたら、自律的に提案する。** 気づくのは Claude の役。
- **だが Claude は勝手に書き込まない。** 提案 → 人間（A・B）が毎回チェック → 承認 で初めて記録する。最終採否は人間。
- 並行作業の最終関門は PR レビュー。
- （思想：機械は発見者でなく処刑人／発見は人間・審判も人間／人気と検証を混ぜない、と同じ構え）

## 参照トリガー（Claude がいつ冷アーカイブを開くか）

冷アーカイブは自動では読まれない。熱の層（CLAUDE.md・メモリ）にあるこの規則に従って、Claude が判断して開く。100%自動ではない＝だから人間が聞ける／提案を承認する多重防御を残す。

**開く（参照する）：**
- 思考レベル：思想・方向性・スコープに関わる判断/提案、新機能の正当化、マネタイズ、法務（金商法・知財）、憲法や核の追加・変更、矛盾の気配、人間が「どうだったっけ」と聞いたとき
- 領域レベル（ルーティング表）：特定の話題の「採否・検証・書き足し」作業に入る前に、対応する書庫を1回見る

| 話題・作業 | 開くファイル |
|---|---|
| 株の法則・検証・SP500 を採否/検証/書き足す | 検証書庫（**最初の発見が出た時に新設**。それまで予約） |
| 法務・知財の判断 | `decisions/` の該当記録 ＋ 必要なら専門家確認の方針 |
| 思想・方向に触れる | `docs/philosophy/00`・`01` |

**開かない（過剰なので参照しない）：**
- 純粋に機械的な作業（コード実装・バグ修正・整形・命名）。雑談で話題が出ただけのとき

線引き：**「手を動かす作業」では見ない。「方向を決める/正当化する/採否する」ときは見る。**

## 改正

この地図そのものを変えるときも、ガバナンスに従う（提案→承認）。土台なので、変更は理由を残して decisions に記録する。
