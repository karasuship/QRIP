# 0002: 開発記録システム（decisions/）の制定
- 状態: 有効
- 日付: 2026-06-20
- 起案: 竹田
- 種別: 設計
- 確認状態: 確認済み

## 背景（経緯）

2人が離れて並行作業し、複雑なプロジェクトを各々が修正・手直し・パッチ追加していく。
竹田の要望：「何を・どういう経緯で・どう変えたかを記録して遡れるようにしたい」。動機は3つ——(1)自分が思い出すため、(2)相手に意図が伝わるため、(3)note・X・YouTube で開発過程を発信し記事にするため。
加えて「新しい記録が古いものと矛盾するなら、古い方を自動で消す／サブに追いやる仕組みを最初に作っておく必要がある」。

## 決定

`decisions/` フォルダに **1決定＝1ファイル（ADR方式）** で開発判断を記録する。

- 各決定は `NNNN-タイトル.md`。テンプレ：状態・日付・起案・種別・確認状態＋背景/決定/理由/廃止したもの。
- **決定は消さない。** 覆ったら古い方の `状態` を `廃止 → 後継: NNNN` にし、`superseded/` へ移す。
- 自動の矛盾検知プログラムは作らない。決定を書く/更新するときにルールを適用し、PRレビューを最終関門にする。
- ルールは `decisions/README.md` に固定。

## 理由

- **「自動で古いものを消す」は採用しなかった。** 理由3つ：(1)地雷②「失敗データを捨てない」に正面から反する——消した瞬間に「遡れる」が壊れる。(2)変遷そのものが一番の発信コンテンツなので、消すと物語が消える。(3)機械が「矛盾」と誤判定して消すのは取り返しがつかない（地雷①⑤の親戚）。
  → 竹田の「サブに追いやる」という直感は正しい。"消す" だけを **"廃止マーク＋ superseded/ へ退避"** に置き換えた。これで矛盾は並存せず（状態欄で今どれが有効か一目）、かつ消さないので Git 履歴＋ファイルの二重で遡れる。
- **1決定＝1ファイル（ADR方式）にした理由**：単一の `DECISIONS.md` に2人が同時追記すると Git で毎回衝突する。別ファイルを足す方式なら並行作業でも衝突しない。
- **重い仕組みを最初から入れない**（地雷⑧の親戚＝オーバーエンジニアリング）。DB やプログラムでなく、Markdown＋Git＋ルール＋Claude の運用で軽く回す。
- これは構築手順書【F】の「書庫」思想（成功も失敗も全部残す／重複検知）の、開発判断版。本体の思想と運用が一致している。

## 廃止したもの
なし（新規制定）。
