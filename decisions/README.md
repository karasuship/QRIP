# 開発記録（Decision Log）

QRIP の「何を・どういう経緯で・なぜ変えたか」を記録し、後から遡れるようにする場所。

## なぜ記録するのか

1. **自分が思い出すため** — 理由を忘れた決定は簡単に覆って、また同じ議論を繰り返す。
2. **相方に意図が伝わるため** — 2人が離れて並行作業するから、文章で経緯を残さないと意図が消える。
3. **発信の素材にするため** — note・X・YouTube で開発過程を発信する。「最初こう決めた→こう覆した→今こう」という変遷そのものが一番のコンテンツ。

これは構築手順書の「書庫」思想（成功も失敗も全部残す／重複検知）の、開発判断版。

## ルール

1. **1決定＝1ファイル**（ADR方式）。`NNNN-タイトル.md`。2人が別々のファイルを足す方式なので、並行作業でもGitで衝突しない。
2. **決定は消さない。** これは絶対。失敗・撤回の記録こそ、判断が本物か偶然かの材料になる（地雷②）。
3. **覆ったら「廃止」にする。** 過去決定を覆す新決定を書いたら、古い方の `状態` を `廃止 → 後継: NNNN` に更新し、ファイルを `superseded/` に移す。なぜ覆ったかを新決定の「廃止したもの」欄に1行残す。
   - これで新旧の矛盾が並存しない（今どれが有効かは `状態` 欄で一目で分かる）。
   - 消さないのでGit履歴＋ファイルの二重で遡れる。
4. **自動で矛盾を消すプログラムは作らない。** 機械の誤判定でデータが消えるリスクが、得られる手間削減を上回る。代わりに、決定を書く/更新するときにこのルールを人間（とClaude）が適用し、PRレビューを最終関門にする。
5. **発信で使える日本語の語りで書く。** 箇条書きの暗号でなく、経緯が読んで分かる文章で。

## 種別

`憲法改正` / `設計` / `実装方針` / `技術選定` / `検証結果`

## 何を記録するか（粒度）

全部をフルADRにすると重くて続かない。逆に何も残さないと「理由を忘れた決定」が起きる。線引き：

- **重い決定 → フルADR（このフォルダに1ファイル）**：思想・設計・技術選定・検証結果・スコープ・マネタイズ・法務など、後から「なぜそうした？」が問われるもの。
- **軽い決定 → 記録不要（Gitのコミットメッセージで十分）**：小さな実装選択・命名・整形・バグ修正など、コードと履歴を見れば分かるもの。

迷ったら「半年後の自分や相方が "なぜ?" と聞きそうか」で判断する。聞きそうなら書く。

## 番号の付け方

- 連番 `NNNN`。2人が同時に同じ番号を取ったら、PRマージ時に後の方がずらして解決する。
- 迷ったら日付を見て最新番号の次を取る。

## テンプレート

```markdown
# NNNN: タイトル
- 状態: 有効
- 日付: YYYY-MM-DD
- 起案: 竹田 / 相方
- 種別: 設計
- 確認状態: 確認済み / 未確認 / 要調査

## 背景（経緯）
なぜこの判断が必要になったか。どんな議論があったか。

## 決定
何を決めた・何を変えたか。

## 理由
なぜそうしたか。他案を退けた理由。

## 廃止したもの
どの過去決定を覆したか・なぜ（あれば）。
```
